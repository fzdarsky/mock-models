## Context

MockLM is a lightweight FastAPI server (port 8080, single `server.py`) that mimics the OpenAI chat completions API and serves as a stand-in for vLLM. Platforms hosting vLLM-based model servers typically include dashboards that query `vllm:`-prefixed Prometheus metrics to display model performance. MockLM currently exposes no metrics, so those dashboard panels show blank or unavailable data.

The server is stateless and single-process by default (`MOCKLM_WORKERS=1`). Token counts are already approximated via word splitting (`estimate_tokens()`). There is no existing instrumentation or metrics dependency.

## Goals / Non-Goals

**Goals:**

- Expose a `/metrics` endpoint on the existing port serving Prometheus text exposition format
- Instrument `/v1/chat/completions` with five vLLM-compatible metrics (request count, latency histogram, in-flight gauge, prompt tokens, generation tokens)
- All metrics carry a `model_name` label derived from `MOCKLM_MODEL_NAME`
- Keep the implementation minimal and contained to a single new module plus light instrumentation in `server.py`

**Non-Goals:**

- Instrumenting other endpoints (`/health`, `/v1/models`, KServe probes)
- OpenTelemetry / OTLP export — most platforms query Prometheus directly
- Custom dashboards or ServiceMonitor/PodMonitor manifests — those are platform-level concerns
- Matching the full vLLM metric surface (dozens of metrics) — only the five most commonly consumed by model-serving dashboards
- Multi-process metric aggregation — MockLM defaults to 1 worker; multi-worker metrics are out of scope

## Decisions

### 1. Use `prometheus-client` library

**Choice:** The standard `prometheus-client` Python package.

**Rationale:** It is the canonical Prometheus client for Python, provides built-in text exposition via `generate_latest()`, supports all needed metric types (Counter, Histogram, Gauge), and has zero transitive dependencies. FastAPI/Starlette integration is trivial — just a `Response` returning the generated bytes.

**Alternatives considered:**

- `starlette-prometheus` / `prometheus-fastapi-instrumentator`: Higher-level wrappers that auto-instrument all routes. Overkill here — we need custom `vllm:`-prefixed metrics on a single endpoint, not generic HTTP metrics.
- Manual text generation: Fragile and pointless when the library exists.

### 2. Dedicated `metrics.py` module

**Choice:** A new `mocklm/src/mocklm/metrics.py` that defines all metric objects and exposes helper functions (`track_request_start`, `track_request_end`).

**Rationale:** Keeps metric definitions and registry management in one place. The server module calls helpers at request boundaries — no metric internals leak into request handling logic.

### 3. Instrument via context manager or explicit start/end calls

**Choice:** Explicit `track_request_start()` / `track_request_end(finish_reason, prompt_tokens, generation_tokens)` calls wrapping the request handler body, plus a `get_metrics_response()` function for the `/metrics` route.

**Rationale:** A context manager would also work, but explicit calls are simpler given that finish reason and token counts are only known after generation completes. The in-flight gauge increments at start, decrements in a `finally` block.

### 4. Metric naming with `vllm:` prefix

**Choice:** Use colon-separated `vllm:` prefix in metric names (e.g., `vllm:request_success_total`).

**Rationale:** Platform dashboards and KServe recording rules query these exact metric names. The colon is valid in Prometheus metric names and conventionally used for recording rules / namespacing. Using `vllm_` underscore prefix would not match the dashboard queries.

### 5. Histogram buckets

**Choice:** Use Prometheus default buckets (0.005 to 10s) for `vllm:e2e_request_latency_seconds`.

**Rationale:** MockLM responses are near-instant (sub-millisecond generation), but configurable delays (`MOCKLM_RESPONSE_DELAY_MS`, `MOCKLM_STREAM_DELAY_MS`) can push latency higher. Default buckets cover the useful range without adding configuration complexity.

## Risks / Trade-offs

- **Multi-worker inaccuracy** → MockLM defaults to 1 worker. With multiple workers, each process maintains its own counters (prometheus-client uses in-process registry). Metrics will be per-worker, not aggregated. This is acceptable for demo purposes and matches the non-goal above.
- **Approximate token counts** → Token metrics use the existing `estimate_tokens()` word-split approximation. This is consistent with how MockLM already reports usage in API responses and is sufficient for dashboard display.
- **Streaming latency measurement** → For streaming requests, end-to-end latency is measured from request receipt to the completion of the full response stream (after all chunks are sent). This reflects real wall-clock time the request occupied.
